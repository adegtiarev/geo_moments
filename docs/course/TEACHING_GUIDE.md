# Teaching Guide

Этот файл задает стандарт для всех следующих уроков. Он нужен, чтобы курс не превращался в список названий технологий.

## Уровень подробности

Каждая новая тема объясняется в формате:

1. Что это такое.
2. Зачем это нужно именно в Geo Moments.
3. Как это связано с уже знакомым Android/backend опытом.
4. Минимальный пример кода.
5. Какой код пишем в проекте.
6. Частые ошибки.
7. Как проверить, что все работает.

Если в практическом задании встречается фраза вроде "добавить `MaterialApp.router`", перед ней должен быть раздел, который объясняет:

- что такое `MaterialApp`;
- чем `MaterialApp.router` отличается от `MaterialApp(home: ...)`;
- какой минимальный код нужен;
- почему мы выбираем этот вариант.

## Формат урока

Каждый урок должен иметь структуру:

```text
# NN Lesson Title

Definition of Done
Что прислать на ревью
```

## Повторение допустимо

Если тема уже встречалась раньше, ее можно кратко повторить. В этом курсе повторение считается частью обучения.

Пример: если в главе 5 снова используется `ConsumerWidget`, нужно дать короткое напоминание:

```text
ConsumerWidget - это StatelessWidget с доступом к WidgetRef.
ref.watch подписывает UI на provider.
```

Не нужно писать "мы уже проходили это в главе 1" вместо объяснения.

## Баланс архитектуры

Курс должен показывать best practices, но не превращать маленькое приложение в enterprise-framework.

Правило:

- если feature маленькая, структура может быть простой;
- если появляется backend, cache, mapping, тесты - разделяем `data/domain/presentation`;
- UI не должен напрямую знать про Supabase/Firebase;
- domain не должен знать про Flutter widgets.

## Проверка после каждой главы

Каждая глава должна завершаться рабочим проектом.

Минимум:

```bash
flutter analyze
flutter test
```

Если глава добавляет platform capability, добавляется ручная проверка:

- auth;
- camera/media;
- map;
- push;
- permissions;
- release build.
